# Module Design & Task List — Outreach OS

A module-based build plan for the sales-engagement system. **No code** — task breakdown only.
Frontend and backend are **separately deployed**; they communicate over a versioned REST/JSON
API (`/api/v1`).

- **Frontend:** React (SPA) — TypeScript, data-fetching layer, component library, state per screen.
- **Backend:** Python **FastAPI** — async services, Pydantic schemas, SQLAlchemy models, workers.
- **Shared contract:** OpenAPI spec generated by FastAPI → typed client consumed by React.

Phase labels: **[P1]** = first milestone (Excel → send → reply → progress), **[P2]** = later.

---

## 0. Foundation (cross-cutting)

### Backend
- [ ] Scaffold FastAPI service: settings/config, app factory, `/api/v1` router mounting, health check.
- [ ] Database layer: Postgres + SQLAlchemy + Alembic migrations baseline.
- [ ] Pydantic schema conventions (request/response models, pagination envelope, error shape).
- [ ] AuthN/AuthZ: session/JWT, role model (Sales/BD, Sales Ops·Admin, Team Lead, System).
- [ ] Background job runtime (Celery/RQ/Arq) + Redis; scheduler for timed jobs.
- [ ] Audit-log service (who/what/when) used by all write endpoints. **[P2]**
- [ ] OpenAPI/Swagger published; spec is the single source of truth for the FE client.

### Frontend
- [ ] Scaffold React app (Vite + TS), routing, layout shell (sidebar nav + topbar — matches prototype).
- [ ] API client generated from OpenAPI; query/caching layer (TanStack Query); auth interceptor.
- [ ] Design system: tokens (colors, type — Hanken Grotesk + IBM Plex Mono), base components
      (Button, Pill/Badge, Toggle, Table, Card, Modal, Toast, RangeSlider, Stepper).
- [ ] Global state: current user/role, workspace, permission gating of nav + actions.
- [ ] Error/empty/loading states convention; route-level guards by role.

## M1 — Data Management  **[P1]**

### Backend
- [ ] Contact & Company models + relations; dedupe keys (email primary; name+company secondary).
- [ ] **Excel/CSV import**: upload endpoint, async parse, column detection, field-mapping apply.
- [ ] Import job lifecycle: preview → validate → commit; partial import + rollback.
- [ ] Validation engine: missing fields, format errors, in-file duplicates, existing-record match (upsert).
- [ ] **Email verification pipeline**: syntax → MX → SMTP probe → catch-all detection; pluggable
      3rd-party provider; status `valid|invalid|risky|unknown` persisted; feeds send-eligibility.
- [ ] Dedupe/merge service: conflict resolution (keep most complete), source-trail retention.
- [ ] Contact/Company profile read endpoints (incl. outreach timeline aggregation).
- [ ] **API incremental import** (upsert) reusing the same data model + dedupe. **[P2]**

### Frontend
- [ ] **Import wizard** (hero): Upload → Map fields → Validate → Verify → Done stepper.
- [ ] Field-mapping UI (table + smart-card variants); required-field enforcement (Email).
- [ ] Validation report screen (counts, issue rows, auto-merge toggle).
- [ ] Email-verification report (pipeline stages, result buckets, exclude-risky toggle, send-eligible count).
- [ ] Contacts list: filters by email status, status chips, pagination/search.
- [ ] Companies list/cards: account-level progress + feasibility status.
- [ ] Contact & Company profile pages with outreach timeline.

## M2 — Campaigns, Templates & Rules  **[P1]**

### Backend
- [ ] **Template model**: subject/body, variables, category, **version**ing.
- [ ] **Template status & edit-lock** (see `Template Status Design.md`): lifecycle `online/offline/draft`;
      `locked = used by ≥1 active campaign`; clone-to-new-version; server-side edit/offline rejection when locked.
- [ ] Variable substitution/render service ( `{first_name}`, `{company}`, `{industry}` … ) + missing-var validation.
- [ ] Campaign model: audience segment, template binding (records launched version), schedule.
- [ ] **Industry rules**: per-industry template/timing/quota/follow-up sequence resolution.
- [ ] **Throttle & quota engine**: per-mailbox daily cap, send interval, batch/drip scheduling, pause/resume/speed.
- [ ] **Suppression engine**: per-contact cool-down, cross-campaign mutual exclusion, blacklist/unsub auto-suppress;
      suppression has top priority over all send rules.
- [ ] **Manual-confirm gate** flag per campaign: off → send queue; on → review queue.
- [ ] Campaign launch/queue endpoints; eligibility computation (valid emails − suppressed).

### Frontend
- [ ] **Templates** screen: cards with status pill + lock badge, usage line, availability toggle
      (Online/Offline), Publish, **Clone to edit**; status legend.
- [ ] Template editor (subject/body, variable insertion, version history). Read-only when locked.
- [ ] **Campaign configuration** screen: audience, template picker (online-only), sending rules
      (throttle slider, window, drip), suppression (cool-down, mutual exclusion, auto-suppress).
- [ ] **Manual-confirm gate** control + explanatory states; launch summary (recipients, throughput, ETA).
- [ ] Campaign list + status (live/review/draft); pause/resume/speed controls.

## M3 — Inbox & Reply Management  **[P1 basic / P2 smart]**

### Backend
- [ ] **Email logging** (outbound): recipient, template+version, send time, status.
- [ ] **Inbox sync** (inbound) from shared mailbox; thread matching via address + `In-Reply-To`/`References`.
- [ ] Reply ↔ contact/campaign association; raw body/subject/thread persisted.
- [ ] **Intent analysis**: P1 rule/keyword classifier; P2 LLM semantic intent
      (`interested / meeting / later / referral / declined / unsubscribe / OOO / bounce`).
- [ ] Routing logic: intent → manual-confirm queue vs auto-follow-up.
- [ ] Manual-confirm queue endpoints (list, actions: send-next / book-meeting / mark-declined / suppress).
- [ ] **Notifications** service (in-app + email/IM e.g. Feishu) configurable by intent tier. **[P2]**
- [ ] **Book team meeting**: scheduling-link send (manual or auto on "meeting" intent); status write-back to timeline. **[P2]**

### Frontend
- [ ] Shared inbox / thread view; outbound + inbound history.
- [ ] Reply list with system-detected intent + suggested action.
- [ ] **Manual-confirm queue** UI with batch/row actions.
- [ ] Notification center / preferences. **[P2]**
- [ ] Meeting-invite action + status on contact timeline. **[P2]**

- [ ] Contact timeline view.

## M5 — Email Infrastructure  **[P1]**

### Backend
- [ ] **Shared mailbox** config (one sending + inbox account); mailbox pool model.
- [ ] **OAuth2 authorization** (Microsoft Graph / Gmail API): auth flow, **refresh-token** storage
      (encrypted, revocable, least-privilege scopes: Mail.Send + Mail.Read). No password/temp tokens.
- [ ] Send adapter over Graph/Gmail; inbox read adapter for sync.
- [ ] **Deliverability**: bounce/complaint/open/reply tracking; SPF/DKIM/DMARC config checks; warm-up ramp.

### Frontend
- [ ] **Mailboxes** screen: shared-account card, OAuth connect/disconnect, provider/auth/scopes.
- [ ] Sending pool + warm-up table (per-mailbox sent/cap, state).
- [ ] Deliverability panel (SPF/DKIM/DMARC checks, bounce/complaint/reply rates).

## Milestone summary

| Milestone | Modules | Goal |
|---|---|---|
| **P1** | M1, M2, M3(basic), M4, M5 | Excel import → verify → throttled send → reply logged → progress query |
| **P2** | M3(intent+meeting+notify), M1 API, M6 | Smart intent routing, auto follow-up, API sync, RBAC + audit |

### Suggested build order
1. Foundation → M5 (mailbox/OAuth, so sends work) → M1 (data + verify) →
2. M2 (templates/status + campaign + throttle/suppression/gate) → M3 basic (logging + inbox sync) →
3. M4 (progress + dashboard) → M3 smart + M6.
