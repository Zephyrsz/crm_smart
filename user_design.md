# 智能 CRM / 销售线索触达系统 — 用户故事与功能需求设计

> 对标 Apollo.io 的内部销售触达（Sales Engagement）系统。核心闭环：
> **数据导入 → 邮箱验证 → 规则化触达 → 回复入库与意图分析 → 人工确认 / 自动跟进 → 进度追踪**
>
> 本文档为需求与设计层，不涉及具体代码实现。

---

## 一、产品定位与范围

一个内部使用的"轻量级智能 CRM + 销售触达"系统，目标是把当前**广发 + 钓鱼（broad-send & fishing）**阶段的冷触达流程标准化、可追踪、可规则化，并在用户回复后引入**意图识别**和**人工确认 / 自动跟进**的分流机制。

**一期定位**：先跑通"Excel 导入 → 群发触达 → 回复入库 → 进度查询"的最小闭环，验证可行性后再扩展 API、自动化和会议预约。

---

## 二、角色定义（Personas）

| 角色 | 职责 | 关注点 |
|---|---|---|
| 销售 / BD | 发起触达、跟进回复、查看进度 | 谁回复了、下一步做什么、别打扰错人 |
| 销售运营 / 管理员 | 导入数据、配置规则与模板、管理邮箱账号 | 数据质量、发送配额、合规 |
| 团队负责人 | 接收会议通知、参与 book meeting | 高意向线索及时升级 |
| 系统（自动化角色） | 定时发送、意图识别、节流、抑制 | 不被判垃圾、不重复打扰 |

---

## 三、功能模块总览

| 模块 | 一句话说明 | 优先级 |
|---|---|---|
| M1 数据管理 | 联系人/公司导入、Profile、邮箱验证、去重 | 一期 |
| M2 触达活动与规则 | 模板、自动触发、行业规则、节流、免打扰 | 一期 |
| M3 收件箱与回复管理 | 邮件入库、意图分析、人工确认、通知、会议 | 一期(基础)+二期(智能) |

| M5 邮件基础设施 | 共享邮箱、OAuth/Token 发送、送达率管理 | 一期 |
| M6 系统与权限 | 角色权限、审计日志 | 二期 |

## 四、用户故事（按模块）

### M1 — 数据管理

**US-1.1 Excel 批量导入（一期）**
作为管理员，我希望上传 Excel 批量导入联系人和公司信息，以便快速建立初始触达名单。
- 支持字段映射（Excel 列 → 系统字段：姓名、邮箱、公司、职位、行业…）
- 导入预览 + 校验报告（缺字段、格式错误、重复行）
- 导入结果可回滚或部分导入

**US-1.2 API 增量导入（后期）**
作为系统集成方，我希望通过 API 新增/更新联系人，以便对接外部数据源做增量同步。
- 与 Excel 导入复用同一套数据模型与去重逻辑
- 支持 upsert（存在则更新，不存在则新增）

**US-1.3 邮箱有效性验证**
作为销售，我希望系统在触达前验证邮箱有效性，以便降低退信、保护发件域名声誉。
- 分层校验：语法 → 域名 MX 记录 → SMTP 探测（含 catch-all 识别）→（可选）第三方验证服务
- 状态标记：`valid` / `invalid` / `risky(catch-all)` / `unknown`
- 验证结果入库并影响是否可发送（见 M2 抑制规则）

**US-1.4 联系人与公司 Profile**
作为销售，我希望查看联系人和公司的完整 Profile，以便判断触达策略。
- 联系人 Profile：基础信息 + 邮箱状态 + 所属公司 + 触达历史时间线
- 公司 Profile：行业、规模、关联联系人列表、整体触达进度

**US-1.5 去重与合并**
作为管理员，我希望系统自动识别并合并重复的联系人/公司，以便避免对同一人多次触达。
- 去重维度：邮箱（主）、姓名+公司（辅）
- 冲突时保留最新/最完整记录，保留来源痕迹

---

### M2 — 触达活动与规则

**US-2.1 邮件模板管理**
作为管理员，我希望创建和管理邮件模板，以便统一话术并支持变量替换。
- 变量占位符（如 `{first_name}` `{company}` `{industry}`）
- 模板按行业/场景分类
- 版本与启用状态管理

**US-2.2 模板邮件自动触发 +「人工确认」开关 ⭐**
作为销售，我希望对一批名单自动触发模板邮件，并可选择是否先经过人工确认，以便在广发效率和质量之间灵活权衡。
- **关键设计决策**：人工确认做成**可配置的开关（gate）**，而非二选一
  - 关闭：直接进入发送队列（广发/钓鱼阶段适用）
  - 开启：进入"待确认队列"，人工逐条/批量审核后才发送
- 触发方式：手动批量触发 / 按计划定时触发

**US-2.3 行业触达规则**
作为管理员，我希望按行业配置触达规则，以便不同行业用不同模板、节奏和频率。
- 规则维度：行业 → 模板、发送时间窗、单日配额、跟进序列
- 初期可只配一套通用规则，后续逐行业细化

**US-2.4 群发节流与配额（防 overwhelm）⭐**
作为管理员，我希望对群发设置节流和配额，以便避免一次群发后回复涌入处理不过来、以及触发垃圾邮件判定。
- 发送侧节流：单邮箱单日发送上限、发送间隔（避免短时间集中发送）
- 处理侧节流：分批投放（如每天 N 封），让回复均匀回流便于人工处理
- 队列可暂停 / 恢复 / 调速

**US-2.5 免打扰与抑制规则（Suppression）⭐**
作为销售，我希望系统自动避免重复打扰同一个人，以便维护品牌形象、符合反骚扰要求。
- **单人免打扰**：同一联系人在触达后进入冷静期，冷静期内不再触发
- **再次触发免打扰**：序列/活动重叠时，同一人只被一个活动命中（互斥）
- **黑名单 / 退订**：已退订、已明确拒绝、无效邮箱自动加入抑制名单，永不发送
- 抑制名单优先级高于一切发送规则

### M3 — 收件箱与回复管理

**US-3.1 邮件入库（收发双向）⭐**
作为销售，我希望所有发出和收到的邮件都进数据库，以便后续查询进度、复盘话术。
- **发件**：每封触达邮件记录（收件人、模板、发送时间、状态）
- **收件**：同步共享收件箱邮件，关联到对应联系人/活动（按邮件地址、In-Reply-To/References 头匹配线程）
- 邮件原文、主题、时间、线程关系均入库

**US-3.2 回复意图自动分析**
作为销售，我希望系统自动分析回复邮件的意图，以便优先处理高价值线索。
- 意图分类（建议枚举）：`有意向/想了解` / `要求安排会议` / `稍后再联系` / `转交他人` / `明确拒绝` / `退订` / `自动回复(OOO)` / `退信`
- 一期可用规则/关键词做轻量分类；二期接入 LLM 做语义意图识别
- 分类结果驱动后续分流（人工确认 or 自动跟进）

**US-3.3 人工确认队列**
作为销售，我希望高意向/不确定的回复进入人工确认队列，以便人工决定下一步。
- 队列展示：联系人、原始回复、系统判定意图、建议动作
- 可执行动作：发送下一步模板 / 安排会议 / 标记拒绝 / 加入免打扰

**US-3.4 通知**
作为销售/负责人，我希望在有人回复（尤其高意向）时收到通知，以便及时跟进。
- 通知渠道：站内 + 邮件 / IM（飞书等）
- 可按意图等级配置（如仅"有意向/要求会议"才通知负责人）

**US-3.5 会议预约（book team meeting）**
作为销售，我希望对高意向回复一键发起团队会议邀约，以便把线索推进到下一阶段。
- 触发方式：人工确认后手动发起，或对"要求安排会议"意图自动发送排期链接
- 关联会议状态回写到联系人时间线

---

### M4 — 触达进度追踪

**US-4.1 特定公司触达进度查询 ⭐**
作为销售，我希望查询某家公司的整体触达进度，以便判断当前是否值得继续投入。
- 展示：**最后联系时间**、最近回复、命中联系人数、整体状态
- **当前可行性**用状态枚举表达（见下方状态机）

**US-4.2 联系人触达时间线**
作为销售，我希望看到单个联系人的完整往来时间线，以便了解上下文。
- 时间线：每次发送、每次回复、状态变更、会议记录，按时间排列

**US-4.3 可行性 / 状态标记**
作为销售，我希望每个联系人/公司都有清晰的状态，以便快速筛选可继续推进的目标。

**建议的状态机（"当前可行性"）：**
```
未触达 → 已触达待回复 → ┬→ 已回复·有意向 → 已安排会议 → 已转化
                        ├→ 已回复·无意向 →（进入免打扰）
                        ├→ 已退订 / 明确拒绝 →（抑制名单）
                        └→ 邮箱无效 / 持续无回复 → 已搁置
```

### M5 — 邮件基础设施

**US-5.1 共享邮箱接入 ⭐**
作为管理员，我希望配置一个共享邮箱作为统一发件 / 收件账号，以便集中管理触达往来。
- 共享邮箱同时承担发送和收件箱同步（M3.1）

**US-5.2 授权发送（OAuth / Token）⭐**
作为管理员，我希望系统能代表共享邮箱自动发送邮件，以便无人值守地跑触达。
- **设计建议**：用 **OAuth2 授权 + refresh token** 的方式（Gmail API / Microsoft Graph），而不是"导出邮件 token"硬塞
  - OAuth2 refresh token 才是可长期、可吊销、可审计的正规方案
  - 你之前用过 Microsoft Graph API 发邮件，这里可直接复用同一套授权与发送通道
  - 避免使用账号密码 / 临时 token，既不稳定也有合规风险
- token 加密存储，权限最小化（只授予发信 + 读收件箱所需 scope）

**US-5.3 送达率管理（Deliverability）**
作为管理员，我希望监控送达健康度，以便避免发件域名进黑名单。
- 监控：退信率、垃圾邮件投诉率、打开/回复率
- 配套：SPF / DKIM / DMARC 配置检查、发送量预热（warm-up）
